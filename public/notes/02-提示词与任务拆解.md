# 提示词与任务拆解

Codex 的输出质量很大程度取决于任务是否清晰。好提示词不是把话写得很长，而是把目标、上下文、限制和验收标准写清楚。

![高质量提示词结构](assets/screenshots/02-prompts/01-prompt-anatomy.webp)

## 推荐结构

一个稳定的 Codex 提示词通常包含 6 个部分：

| 部分 | 作用 | 示例 |
| --- | --- | --- |
| 背景 | 帮 Codex 定位项目和技术栈 | “这是 React + Vite 项目，样式用 CSS Modules” |
| 目标 | 明确要完成什么 | “修复登录按钮在 390px 宽度下被遮挡” |
| 范围 | 限制修改边界 | “只改 Login 组件和样式，不改 API” |
| 约束 | 避免不希望的实现 | “不要引入新依赖，不要重写整个页面” |
| 验收 | 定义完成标准 | “移动端和桌面都不溢出，npm test 通过” |
| 输出 | 要求最终说明格式 | “说明改了哪些文件、跑了哪些命令、还有哪些风险” |

## 复杂任务先计划

涉及跨模块改动、迁移、重构、权限、数据删除、外部服务时，先要求 Codex 计划，不要直接执行。

![复杂任务先计划](assets/screenshots/02-prompts/02-plan-first.webp)

推荐提示词：

```text
这是一个复杂改动。请先只读分析，不要修改文件。

请输出：
1. 你需要阅读哪些文件；
2. 可能的实现方案；
3. 每个方案的风险；
4. 推荐方案；
5. 预计会改哪些文件；
6. 需要运行哪些验证命令。

等我确认后再开始改文件。
```

## 把大需求拆成小任务

不推荐：

```text
帮我把后台系统优化一下，顺便修一下页面、接口、权限和性能问题。
```

推荐拆分：

```text
第一步：只读分析后台系统的导航和路由结构，找出设置页相关文件，不要改文件。
```

```text
第二步：只修设置页移动端布局问题，不改接口和权限逻辑。完成后运行前端测试。
```

```text
第三步：审查这次改动是否影响其他页面，重点看 CSS 选择器和共享组件。
```

这种拆法有三个好处：

- 每次 diff 更小，容易审查。
- 出错时容易定位是哪一步引入问题。
- Codex 更容易选择合适的工具和验证方式。

## 常用提示词模板

### 只读理解代码

```text
请只读分析这个模块，不要修改文件。
请解释：
1. 入口在哪里；
2. 数据流如何流动；
3. 关键函数分别负责什么；
4. 这个模块有哪些潜在风险或测试缺口。
```

### 修复 bug

```text
Bug：点击保存后 UI 显示成功，但刷新页面后设置没有保留。

请：
1. 先定位可能原因；
2. 列出你准备检查的文件；
3. 修改最小必要代码；
4. 添加或更新测试；
5. 运行相关测试；
6. 最终说明根因、改动和验证结果。
```

### 前端页面修改

```text
目标：修复 /settings 在 390px 宽度下按钮溢出的问题。

约束：
- 不改接口；
- 不引入新 UI 库；
- 保持桌面端布局不退化；
- 只改相关组件和样式。

验证：
- 请启动或检查 dev server；
- 用内置浏览器查看 /settings；
- 检查桌面和移动宽度；
- 最终附上验证结果。
```

### 代码审查

```text
请以代码审查姿态检查当前未提交改动。
重点关注：
1. 可能的 bug 或回归；
2. 边界条件；
3. 错误处理；
4. 是否缺少测试；
5. 是否混入无关改动。

请按严重程度排序输出发现，并引用文件和行号。
```

### 文档生成

```text
请为这个功能生成中文使用文档。
要求：
- 面向第一次接触项目的开发者；
- 包含步骤、示例、常见坑和检查清单；
- 不要凭空编造未在代码或官方文档中出现的能力；
- 文末列出参考来源。
```

## 什么信息不要直接给 Codex

除非任务确实需要，且你清楚数据会被如何使用，否则不要在提示词里贴：

- 密码、密钥、Token、Cookie。
- 真实身份证件、财务、医疗、合同等敏感信息。
- 公司内部不能外发的数据。
- 未脱敏的用户数据、日志、崩溃报告。

如果需要分析日志，先脱敏；如果需要连接私有系统，优先使用经过授权的 Connector / MCP，而不是把大量私有数据直接粘贴进聊天。

## 让最终结果可验收

你可以要求 Codex 最终必须回答：

```text
最终请按这个格式回复：
1. 改了什么；
2. 改了哪些文件；
3. 运行了哪些验证命令；
4. 结果是否通过；
5. 还有哪些残留风险。
```

这样做的价值是：你不用从长篇聊天里猜它到底做了什么。

## 好物推荐：让提示词少写一半

很多“提示词技巧”最后都应该沉淀成工具，而不是永远靠复制粘贴。

| 场景 | 推荐 | 为什么能提效 | 推荐触发方式 |
| --- | --- | --- | --- |
| 经常写同一类任务说明 | `$skill-creator` | 把提示词模板、步骤、验收标准固化为 Skill | “请把这个稳定流程做成一个 Skill” |
| 不知道有哪些现成 Skill | `$skill-installer` | 安装 curated skills 或来自仓库的 skills | “帮我找适合数据分析/文档生成的 skill” |
| 需要核对官方文档 | OpenAI Docs MCP / 文档类 MCP | 让 Codex 先查资料再回答，降低过时风险 | “先查官方文档再给方案” |
| 需要并行调研 | Subagents / 多线程 | 多个方向同时探索，再汇总结论 | “开多个方向分别调研，然后汇总推荐” |
| 需要团队统一格式 | 自定义 Skill + AGENTS.md | Skill 管流程，AGENTS.md 管项目规则 | “按项目 AGENTS.md 和 review skill 输出” |

值得自制的 Skill：

- **bug-triage**：固定要求复现步骤、根因、最小修复、回归测试。
- **frontend-visual-qa**：固定要求检查视口、长文本、空状态、错误状态。
- **review-checklist**：固定输出严重程度、文件行号、风险、建议修复。
- **release-notes**：从 commit 或 PR 中生成变更日志。
- **docs-from-code**：从代码和 README 生成面向新人的文档。

不建议做成 Skill 的内容：

- 今天一次性的需求。
- 还没稳定的探索流程。
- 需要经常变化的项目事实，例如临时分支名、一次性测试数据。

## 检查清单

- [ ] 目标是否是一句话能说清的结果。
- [ ] 是否说明了“不改什么”。
- [ ] 是否说明了验收标准。
- [ ] 是否要求必要时先计划。
- [ ] 是否要求最终给出验证结果。
- [ ] 是否避免了敏感信息。
