# 代码修改、测试与审查

Codex 能写代码，但你仍然需要把它的输出当作候选补丁来审查。可靠的工作流应该包含：明确范围、小步修改、运行验证、查看 diff、记录残留风险。

![使用 Review 面板看改动](assets/screenshots/04-review-test/01-review-pane.webp)

## 修改前：先收窄范围

在让 Codex 修改代码前，先说清楚：

- 允许改哪些文件或目录。
- 不允许改哪些文件或目录。
- 是否允许新增依赖。
- 是否需要新增测试。
- 验证命令是什么。

示例：

```text
请修复 src/pages/settings 中的移动端布局问题。
范围限制：
- 只改 settings 页面相关组件和样式；
- 不改接口、不改路由、不新增依赖；
- 如果需要改共享组件，先说明理由并等我确认。

完成后请运行相关测试，并在最终说明中写出命令和结果。
```

## 修改中：让 Codex 小步提交思路

复杂修改可以要求 Codex 中途说明：

```text
请先定位问题并说明根因。
如果确认需要修改多个文件，请先列出文件清单和原因。
不要一次性做大范围重构。
```

这能避免 Codex 把一个小 bug 扩展成大型重构。

## 修改后：必须看 diff

Codex app 的 Review / diff 面板可以查看变更文件和逐行修改。审查时重点看：

- 是否改了无关文件。
- 是否引入新依赖或锁文件变化。
- 是否删除了重要错误处理。
- 是否修改了公共 API 或数据结构。
- 是否把临时代码、日志、调试输出留在代码里。
- 是否新增或更新了测试。

审查提示词：

```text
请审查你刚才的改动。
重点检查：
1. 是否有无关文件；
2. 是否可能引入回归；
3. 是否缺少测试；
4. 是否有错误处理或边界条件遗漏；
5. 是否有临时代码或调试输出。
```

## 运行验证命令

验证命令取决于项目。常见命令包括：

| 项目类型 | 常见命令 |
| --- | --- |
| Node / 前端 | `npm test`、`npm run lint`、`npm run build` |
| pnpm 项目 | `pnpm test`、`pnpm lint`、`pnpm build` |
| Python | `pytest`、`ruff check`、`mypy` |
| .NET | `dotnet test`、`dotnet build` |
| 文档项目 | 链接检查、Markdown lint、导出预览 |

![运行验证命令](assets/screenshots/04-review-test/02-run-tests.webp)

要求 Codex 记录验证结果：

```text
请运行最相关的验证命令。
最终回复必须包含：
- 命令；
- 是否通过；
- 如果失败，失败原因和你是否已修复；
- 没有运行的命令及原因。
```

## 测试失败怎么办

测试失败不一定意味着 Codex 做错了，也可能是环境缺依赖、已有测试失败或命令不适合当前平台。处理顺序：

1. 让 Codex 保留失败输出，不要马上大改。
2. 判断失败是否和本次修改相关。
3. 如果相关，要求最小修复。
4. 如果无关，让 Codex 在最终说明里标注“已有失败”或“环境问题”。
5. 不要为了让测试通过而删除测试或降低断言，除非你明确要求。

提示词：

```text
测试失败后请不要直接删除或弱化测试。
请先判断失败是否由本次改动引起，并给出证据。
如果是本次改动导致，请做最小修复。
如果不是，请说明原因并停止。
```

## 代码审查姿态

当你明确说“review”或“审查”时，可以要求 Codex 采用审查输出格式：

```text
请以代码审查姿态检查当前 diff。
先列问题，按严重程度排序。
每个问题请包含：
- 文件和行号；
- 为什么这是问题；
- 可能影响；
- 建议修复方式。
如果没有发现问题，请说明剩余风险和未运行的验证。
```

## 最终说明应该包含什么

优秀的最终说明应该短而完整：

```text
已完成：
- 修复 SettingsToolbar 在窄屏下按钮溢出。
- 将按钮容器改为可换行布局，并保留桌面端间距。
- 新增移动端回归测试。

验证：
- npm test：通过。
- npm run lint：通过。

风险：
- 未在真实移动设备上验证，只通过浏览器视口检查。
```

## 常见错误

**错误：让 Codex 改完后直接提交。**  
更好的做法：先看 diff 和验证输出，再决定是否提交。

**错误：测试失败后让 Codex “修到通过”为止。**  
更好的做法：要求先判断失败是否相关，避免无关改动扩大。

**错误：让 Codex 同时“重构并修 bug”。**  
更好的做法：先修 bug，再单独开线程讨论重构。

**错误：没有写验收标准。**  
更好的做法：明确通过什么测试、什么页面状态、什么行为算完成。

## 好物推荐：代码质量和审查效率

代码修改类任务最值得投资的是“验证工具”和“审查工具”，而不是更多聊天模板。

| 推荐 | 类型 | 提升什么效率 | 适合场景 |
| --- | --- | --- | --- |
| GitHub 集成 / GitHub MCP | MCP / App | 读取 PR、issue、CI、review 上下文 | 团队在 GitHub 协作 |
| Codex Security 插件 | Plugin | 对授权代码做安全扫描和漏洞验证 | 安全敏感仓库、上线前审查 |
| review-checklist skill | 自定义 Skill | 固定审查输出格式和团队风险偏好 | 每个 PR 都要审一遍 |
| test-plan skill | 自定义 Skill | 根据改动范围选择最小测试集 | 大仓库、测试很慢的项目 |
| Browser / in-app browser | Plugin / 内置 | 让前端改动有真实渲染证据 | UI、CSS、交互、文档站 |
| Jupyter / Spreadsheets skill | Skill | 对测试结果、性能数据做分析 | 性能回归、数据验证 |

高收益的自制 Skill：

```text
review-checklist:
- 先列 P0/P1/P2 问题；
- 必须引用文件和行号；
- 优先找 bug、回归、缺失测试；
- 总结放最后；
- 没发现问题时说明残留风险。
```

```text
test-plan:
- 根据 git diff 判断改动类型；
- 先推荐最小验证命令；
- 对共享模块和公共 API 推荐扩展测试；
- 如果命令失败，判断是否和本次改动相关。
```

什么时候不该用重型工具：

- 只改一行文案，不需要 Codex Security 扫全仓。
- 只解释代码，不需要 GitHub 写权限。
- 小型 CSS 修复，先用 in-app browser 看页面即可，不必接 Chrome 登录态。

## 检查清单

- [ ] 修改范围足够小。
- [ ] diff 中没有无关文件。
- [ ] 没有新增未批准依赖。
- [ ] 已运行相关测试或说明无法运行原因。
- [ ] 最终说明包含改动、验证和风险。
- [ ] 重要变更已人工审查。
